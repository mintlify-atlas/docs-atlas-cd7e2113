The contracts workspace CLI is a separate package focused on filesystem-based contract repository management.

**Package:** `@open-agreements/contracts-workspace`  
**Binary:** `open-agreements-workspace`

## Why Separate from open-agreements?

`open-agreements` is optimized for template/recipe filling. The workspace CLI handles repository-level workflows:

- Topic-first structure planning and suggestions
- Forms catalog validation and download (URL + SHA-256 checksum)
- Filename-driven execution status
- YAML status indexing and linting

Teams can use one package without requiring the other.

## Commands

### init

Preview a workspace setup in the current directory. This command **does not** create folders/files automatically.

<CodeGroup>
```bash Basic Usage
open-agreements-workspace init
```

```bash With Agent Snippets
open-agreements-workspace init --agents claude,gemini
```

```bash With Custom Topics
open-agreements-workspace init --topics employment,vendor,customer
```
</CodeGroup>

**What it suggests:**

- Top-level topic folders (e.g., `employment/`, `vendor/`, `customer/`)
- Lifecycle subfolders (`draft/`, `executed/`)
- Collaboration files:
  - `CONTRACTS.md` — Overview and conventions
  - `forms-catalog.yaml` — Template inventory with checksums
  - `WORKSPACE.md` — Workspace structure documentation
  - `.contracts-workspace/conventions.yaml` — Naming and structure config
  - `contracts-index.yaml` — Generated status index (via `status generate`)
- Optional agent snippets under `.contracts-workspace/agents/`

<Note>
  The `init` command only **suggests** folders and files. You create them manually or via the suggested shell commands.
</Note>

### catalog validate

Validate forms catalog schema and required checksum fields.

<CodeGroup>
```bash Default Path
open-agreements-workspace catalog validate
```

```bash Custom Path
open-agreements-workspace catalog validate --catalog-path path/to/forms-catalog.yaml
```
</CodeGroup>

**Validation checks:**

- YAML schema compliance
- Required fields: `id`, `url`, `sha256`
- Valid SHA-256 format (64 hex chars)
- Unique entry IDs

### catalog fetch

Fetch allowed entries and verify SHA-256 before writing files.

<CodeGroup>
```bash Fetch All Entries
open-agreements-workspace catalog fetch
```

```bash Fetch Specific Entries
open-agreements-workspace catalog fetch yc-safe-valuation-cap yc-safe-mfn
```
</CodeGroup>

**Behavior:**

- Downloads files to `forms/` directory in workspace root
- Verifies SHA-256 checksum before writing
- Skips pointer-only/proprietary entries (reports but does not vendor)
- Fails if checksum mismatch

<Warning>
  Only entries marked as `redistributable: true` are fetched automatically. Pointer-only entries require manual download.
</Warning>

### status generate

Generate `contracts-index.yaml` with timestamped summary and per-document records.

<CodeGroup>
```bash Default Output
open-agreements-workspace status generate
```

```bash Custom Output Path
open-agreements-workspace status generate --output-path custom-index.yaml
```
</CodeGroup>

**Generated index includes:**

- Timestamp of generation
- Total document count
- Draft vs. executed counts
- Per-document records with:
  - File path
  - Lifecycle stage (draft/executed)
  - Detected naming conventions
  - Topic classification
- Lint findings

### status lint

Validate structure and naming conventions.

```bash
open-agreements-workspace status lint
```

**Lint checks:**

- Required lifecycle folders exist
- Disallowed file types by folder (configurable)
- Executed marker naming consistency (uses configured marker, default `_executed`)
- Stale `contracts-index.yaml` detection
- Duplicate/timestamped file detection
- Root orphan detection (files not in topic folders)
- Cross-contamination detection (file in wrong domain folder)

## Convention Config

Workspace naming and structure conventions are stored in `.contracts-workspace/conventions.yaml`.

### Adaptive Init

When `init` runs on a non-empty directory, it scans existing filenames to infer:

- **Executed marker**: Detects `_executed`, `_signed`, `(fully executed)`, `(signed)` patterns
- **Naming style**: Detects `snake_case`, `kebab-case`, `title-case-spaces`, `title-case-dash`
- **Domain applicability**: Classifies folders as lifecycle-applicable or asset-only

On an empty directory, defaults apply (`snake_case`, `_executed`).

### Generated Docs (WORKSPACE.md / FOLDER.md)

`init` generates `WORKSPACE.md` at the workspace root and `FOLDER.md` in each existing lifecycle folder. These files:

- Are convention-aware (reflect configured naming style and markers)
- Use sentinel comments for idempotent re-generation
- Preserve user content added outside the sentinel block on re-init

<Tip>
  Re-running `init` updates generated sections while preserving your custom notes.
</Tip>

## Local-Synced Drive Model (v1)

This CLI is filesystem-only in v1. It is designed to work on local folders, including locally synced cloud-drive folders (e.g., Google Drive sync on macOS).

**No Drive API required:**

- Works on local filesystem paths
- Compatible with Google Drive File Stream, Dropbox, OneDrive, etc.
- No OAuth credentials needed
- No cloud API rate limits

<Check>
  Just point the CLI at your synced folder path and it operates on the local files.
</Check>

## Out of Scope (v1)

- Signature request integrations (DocuSign, HelloSign, etc.)
- Automatic PDF splitting/signature-pack processing
- Cloud-native document API orchestration
- Real-time collaboration features

## Demo Workflow

<Steps>
  <Step title="Initialize Workspace">
    ```bash
    cd /Users/you/contracts
    open-agreements-workspace init --agents claude,gemini
    ```
    Review suggested folders and files.
  </Step>
  <Step title="Create Suggested Structure">
    ```bash
    mkdir -p employment/draft employment/executed vendor/draft vendor/executed
    touch CONTRACTS.md forms-catalog.yaml WORKSPACE.md
    ```
  </Step>
  <Step title="Validate Catalog">
    ```bash
    open-agreements-workspace catalog validate
    ```
  </Step>
  <Step title="Fetch Catalog Entries">
    ```bash
    open-agreements-workspace catalog fetch
    ```
    Downloads redistributable templates to `forms/` directory.
  </Step>
  <Step title="Add Sample File">
    ```bash
    # Add a sample contract to employment/draft/
    cp sample-offer-letter.docx employment/draft/
    ```
  </Step>
  <Step title="Generate Status Index">
    ```bash
    open-agreements-workspace status generate
    ```
    Creates `contracts-index.yaml` with document inventory.
  </Step>
  <Step title="Lint Workspace">
    ```bash
    open-agreements-workspace status lint
    ```
    Detects naming inconsistencies, missing folders, etc.
  </Step>
</Steps>

├── .contracts-workspace/

## Naming Conventions

### Executed Marker

The workspace detects executed documents by marker suffixes:

- `_executed` (default)
- `_signed`
- `(fully executed)`
- `(signed)`

**Example:**

```
employment/draft/offer_letter_jane_doe.docx
employment/executed/offer_letter_jane_doe_executed.docx
```

### Naming Styles

- `snake_case` — `offer_letter_jane_doe.docx` (default)
- `kebab-case` — `offer-letter-jane-doe.docx`
- `title-case-spaces` — `Offer Letter Jane Doe.docx`
- `title-case-dash` — `Offer-Letter-Jane-Doe.docx`

<Note>
  The CLI infers your preferred style from existing files and applies it to generated documentation.
</Note>

## Lint Rules

### Required Folders

Ensures lifecycle folders exist for each topic:

- `{topic}/draft/`
- `{topic}/executed/`

### Executed Marker Consistency

Verifies all executed documents use the configured marker.

### Stale Index Detection

Warns if `contracts-index.yaml` is older than the most recently modified document.

### Duplicate Detection

Identifies files with duplicate base names (ignoring timestamps and executed markers).

### Root Orphan Detection

Reports documents in the workspace root that should be in topic folders.

### Cross-Contamination

Detects documents in the wrong topic folder based on filename analysis.
